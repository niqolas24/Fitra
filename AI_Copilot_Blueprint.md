# AI Application Copilot — Complete Technical Blueprint
### For University Hackathon MVP

---

## 1. Product Definition

**AI Application Copilot** is a web-based tool that helps university students evaluate how well their resume matches a specific job or internship description, then gives structured, honest, actionable feedback to improve their chances.

**Core problem it solves:**
Students apply blindly. They don't know if they're a real fit, which keywords matter, what's missing from their resume, or how to tailor it without lying. ATS systems silently filter them out before a human ever reads their resume.

**What makes it different from generic resume checkers:**
- It does not just scan for buzz words generically. It analyzes *your resume against a specific job posting*.
- It uses a hybrid scoring engine (deterministic logic + fuzzy matching + LLM reasoning), not just an LLM guessing a score.
- It never invents qualifications. Every suggestion is grounded in what the student already has.
- It gives section-by-section feedback, not a single vague verdict.

**Who uses it:** University students applying to internships and entry-level roles in tech, finance, consulting, and similar fields.

## 2. MVP Scope (Hackathon)

Build exactly these features. Cut everything else.

### In Scope
- Resume upload: PDF and DOCX → raw text extraction + section splitting
- Job description: paste-in text field
- Keyword extraction from JD: must-have, nice-to-have, technical, soft skills
- Keyword matching: matched vs. missing, with fuzzy matching
- Hybrid fit score (0–100) + label (Strong / Moderate / Weak Fit)
- Score breakdown panel showing component weights
- Red flag detection (8+ categories, rule-based + LLM)
- ATS formatting warnings (rule-based)
- Tailoring suggestions (LLM, truthful only)
- Results dashboard UI showing all outputs
- Health check endpoint

### Out of Scope for Hackathon
- User accounts / authentication
- Persistent database storage
- Resume editing in-browser
- Cover letter generation
- Multi-resume comparison
- Interview prep features

## 3. System Architecture

```
┌─────────────────────────────────────────────────────┐
│                   NEXT.JS FRONTEND                   │
│  Upload UI → JD Input → Loading State → Dashboard   │
└────────────────────────┬────────────────────────────┘
                         │ HTTP (multipart form + JSON)
                         ▼
┌─────────────────────────────────────────────────────┐
│                  FASTAPI BACKEND                     │
│                                                      │
│  POST /api/analyze                                   │
│       │                                              │
│       ├─ 1. ResumeParser                            │
│       │      PDF (pdfplumber) | DOCX (python-docx)  │
│       │      → raw text + section dict              │
│       │                                              │
│       ├─ 2. JDExtractor (LLM)                       │
│       │      → structured JSON: skills, tools,      │
│       │        keywords, responsibilities, quals     │
│       │                                              │
│       ├─ 3. KeywordMatcher (RapidFuzz)              │
│       │      → matched[], missing[], scores          │
│       │                                              │
│       ├─ 4. FitScorer (deterministic)               │
│       │      → score (0-100), label, breakdown      │
│       │                                              │
│       ├─ 5. ATSChecker (rule-based)                 │
│       │      → ats_warnings[]                        │
│       │                                              │
│       └─ 6. LLMAnalyzer (OpenAI)                    │
│              → red_flags, tailoring_suggestions,    │
│                fit_explanation, bullet_rewrites      │
│                                                      │
│       → Assemble AnalysisResponse → return JSON     │
└─────────────────────────────────────────────────────┘
                         │
                         ▼
              OpenAI API (gpt-4o or gpt-4o-mini)
```

### Pipeline Design Principle
Each step is a separate service. The LLM is called **twice** with focused prompts:
1. First call: extract structured data from the JD
2. Second call: generate red flags + tailoring suggestions with full context

The score is **never** generated by the LLM. It is computed deterministically from matching outputs.

## 5. Backend API Design

### Endpoints

#### `POST /api/analyze`
**Content-Type:** `multipart/form-data`

**Request fields:**
- `resume` — File (PDF or DOCX, max 10MB)
- `job_description` — string (pasted JD text)

**Response:** `AnalysisResponse` JSON (see schemas below)

**Errors:**
- `422` — validation error (missing fields, wrong file type)
- `400` — file parsing failed
- `500` — upstream LLM error

#### `GET /api/health`
Returns `{ "status": "ok", "version": "1.0.0" }`

### Pydantic Schemas

```python
# Request
class AnalyzeRequest(BaseModel):
    job_description: str

# Keyword entry
class Keyword(BaseModel):
    term: str
    category: Literal["must_have", "nice_to_have", "technical", "soft"]
    matched: bool
    match_score: Optional[float] = None     # 0-100, from fuzzy match
    matched_as: Optional[str] = None        # what it matched in resume

# Red flag
class RedFlag(BaseModel):
    type: str                               # e.g. "vague_bullet"
    description: str
    severity: Literal["high", "medium", "low"]
    location: Optional[str] = None          # e.g. "Experience > Software Engineer"

# ATS warning
class ATSWarning(BaseModel):
    type: str                               # e.g. "multi_column_suspected"
    description: str
    suggestion: str

# Bullet rewrite suggestion
class BulletRewrite(BaseModel):
    original: str
    rewritten: str
    reason: str

# Tailoring suggestion
class TailoringSuggestion(BaseModel):
    area: str                               # e.g. "Projects", "Skills"
    suggestion: str
    keywords_to_add: List[str]
    bullet_rewrites: Optional[List[BulletRewrite]] = []

# Fit score breakdown
class ScoreBreakdown(BaseModel):
    keyword_coverage: float                 # 0-40
    must_have_coverage: float               # weighted sub-component
    experience_alignment: float             # 0-30, from LLM structured score
    project_relevance: float                # 0-20, from LLM structured score
    red_flag_penalty: float                 # negative, max -20

# Full analysis response
class AnalysisResponse(BaseModel):
    fit_score: int                          # 0-100
    fit_label: Literal["strong_fit", "moderate_fit", "weak_fit"]
    fit_explanation: str                    # LLM-generated paragraph
    score_breakdown: ScoreBreakdown
    keywords: List[Keyword]
    matched_count: int
    missing_count: int
    resume_sections: Dict[str, str]         # section name → text
    red_flags: List[RedFlag]
    ats_warnings: List[ATSWarning]
    tailoring_suggestions: List[TailoringSuggestion]
    jd_summary: Dict                        # extracted JD fields
    processing_time_ms: int
```

## 6. Rule-Based vs LLM-Based

| Feature | Approach | Why |
|---|---|---|
| Resume text extraction | Rule-based (pdfplumber, python-docx) | Deterministic, no API cost |
| Section splitting | Rule-based (regex heading detection) | Fast, reliable |
| JD keyword extraction | **LLM** (structured JSON output) | JDs vary too much in structure for regex |
| Keyword matching | Rule-based (RapidFuzz token_sort_ratio) | Deterministic, fuzzy handles acronyms |
| Fit score computation | Rule-based (weighted formula) | Score must be explainable, not vibes |
| ATS formatting checks | Rule-based (regex + text analysis) | Objective formatting rules |
| Weak action verb detection | Rule-based (blocklist) | Fast, consistent |
| Inconsistent tense detection | Rule-based (spaCy POS tagging) | Objective grammar rule |
| Vague bullet detection | **LLM** | Requires contextual judgment |
| Missing measurable impact | **LLM** | Requires semantic understanding |
| Role alignment red flags | **LLM** | Requires reasoning |
| Tailoring suggestions | **LLM** | Requires creative rewriting within constraints |
| Bullet rewrites | **LLM** | Language generation task |
| Fit explanation | **LLM** | Narrative summary of computed data |
| Experience alignment score | **LLM** (returns number 0–100) | Provides sub-score fed into deterministic scorer |
| Project relevance score | **LLM** (returns number 0–100) | Same |

### Key principle: LLM outputs numbers, rules use them
The LLM returns `experience_alignment_score` (0–100) and `project_relevance_score` (0–100) as part of its structured JSON. The deterministic scorer then uses those numbers as weighted components. This makes the final score reproducible and auditable.

## 7. Database Schema (Optional — PostgreSQL / Supabase)

If you add persistence, use these tables:

```sql
-- Users (if auth added later)
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email TEXT UNIQUE NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now()
);

-- Stored resumes
CREATE TABLE resumes (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES users(id),
    filename TEXT NOT NULL,
    raw_text TEXT NOT NULL,
    sections JSONB,                    -- {"Experience": "...", "Skills": "..."}
    created_at TIMESTAMPTZ DEFAULT now()
);

-- Stored job descriptions
CREATE TABLE job_descriptions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES users(id),
    raw_text TEXT NOT NULL,
    extracted_data JSONB,             -- full JD extraction result
    created_at TIMESTAMPTZ DEFAULT now()
);

-- Analysis results
CREATE TABLE analyses (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    resume_id UUID REFERENCES resumes(id),
    job_description_id UUID REFERENCES job_descriptions(id),
    fit_score INTEGER NOT NULL,
    fit_label TEXT NOT NULL,
    full_result JSONB NOT NULL,        -- full AnalysisResponse
    created_at TIMESTAMPTZ DEFAULT now()
);
```

**For hackathon:** Skip the database entirely. Return results synchronously and let the frontend hold state in memory. Add a "Download Report" button as a PDF export if you want persistence without a DB.

## 9. LLM Prompts

### Prompt 1: Job Description Keyword Extraction

```
You are an expert technical recruiter and NLP specialist.

Analyze the following job description and extract structured information.
Return ONLY valid JSON matching the schema below. Do not include any text before or after the JSON.

SCHEMA:
{
  "role_title": string,
  "company_type": string,  // e.g. "startup", "FAANG", "consulting firm"
  "required_skills": [string],       // explicitly required
  "preferred_skills": [string],      // nice to have, preferred, bonus
  "technical_tools": [string],       // specific software, frameworks, languages, platforms
  "soft_skills": [string],           // communication, leadership, etc.
  "action_verbs": [string],          // verbs used in responsibilities section
  "key_responsibilities": [string],  // 5-8 main job duties
  "qualifications": [string],        // education/experience requirements
  "must_have_keywords": [string],    // top 10 terms a resume MUST contain to pass ATS
  "nice_to_have_keywords": [string], // useful but not disqualifying if absent
  "experience_alignment_criteria": string, // what kind of experience best fits this role
  "project_relevance_criteria": string     // what kind of projects would impress
}

JOB DESCRIPTION:
{job_description}
```

### Prompt 2: Resume Red Flag Analysis + Experience/Project Scoring

```
You are an expert resume reviewer helping a university student applying for the following role.

Your job is to:
1. Identify specific red flags in the resume
2. Score how well the student's EXPERIENCE aligns with the role (0-100)
3. Score how relevant the student's PROJECTS are to the role (0-100)

Return ONLY valid JSON matching this schema:

{
  "experience_alignment_score": integer,  // 0-100
  "project_relevance_score": integer,     // 0-100
  "red_flags": [
    {
      "type": string,         // one of: "vague_bullet", "no_impact_metrics", "weak_action_verb",
                              //   "inconsistent_tense", "role_mismatch", "generic_skills_section",
                              //   "no_outcome_shown", "short_tenure_unexplained", "other"
      "description": string,  // specific and concrete, reference the actual resume text
      "severity": "high" | "medium" | "low",
      "location": string      // which section/bullet this was found in
    }
  ]
}

SCORING GUIDE:
- experience_alignment_score: 0=no relevant experience, 50=some overlap, 80+=strong match
- project_relevance_score: 0=no relevant projects, 50=tangentially relevant, 80+=directly relevant

RED FLAG RULES:
- Only flag REAL issues visible in the resume text. Do not invent problems.
- Be specific. "The bullet 'Worked on backend systems' has no metrics and no outcome." is good.
- "Vague bullet" is bad feedback. Good: "bullet starting with 'Assisted with...' is vague and passive."
- Flag a maximum of 8 red flags. Prioritize high severity.
- Do NOT flag missing keywords (that is handled separately).

JOB DESCRIPTION CONTEXT:
Role: {role_title}
Key Requirements: {key_requirements}

RESUME TEXT:
{resume_text}
```

### Prompt 3: Truthful Tailoring Suggestions + Bullet Rewrites

```
You are an expert career coach helping a university student tailor their resume for a specific job.

CRITICAL CONSTRAINT: You MUST NOT invent, fabricate, or imply experience the student does not have.
Every suggestion must be grounded in what is already in their resume.

Your job:
1. Suggest which existing experiences/projects to emphasize
2. Rewrite up to 5 specific resume bullets to better align with the job (stronger verbs, more specific, add context)
3. Recommend where existing keywords could be naturally added

Return ONLY valid JSON:

{
  "tailoring_suggestions": [
    {
      "area": string,              // e.g. "Experience > Software Engineer at XYZ", "Projects", "Skills"
      "suggestion": string,        // specific, actionable advice
      "keywords_to_add": [string], // from the missing keywords that actually apply to this area
      "bullet_rewrites": [
        {
          "original": string,      // copy the exact original bullet text
          "rewritten": string,     // improved version — same facts, stronger framing
          "reason": string         // why this rewrite is better
        }
      ]
    }
  ]
}

RULES FOR BULLET REWRITES:
- Do not add tools, technologies, or skills not mentioned in the original
- Use stronger action verbs where the original is passive or weak
- Add quantification ONLY if the context strongly implies a specific number (e.g. "a team" → "a team of 4")
- If you cannot improve a bullet honestly, skip it
- Format: [Strong Verb] + [What you did] + [How/With what] + [Result/Impact]

MISSING KEYWORDS (from this job's requirements that aren't in the resume):
{missing_keywords}

JOB REQUIREMENTS SUMMARY:
{jd_summary}

RESUME FULL TEXT:
{resume_text}
```

### Prompt 4: Fit Explanation (Short, After Score is Computed)

```
You are writing the "Fit Summary" section of an AI application copilot report.

The student has already received a computed fit score. Your job is to write a clear,
honest, 3-4 sentence explanation of why they received this score and what the most
important gaps or strengths are.

Be direct. Be truthful. Do not be excessively encouraging or discouraging.
Do not recommend the student apply or not apply — just explain the score.

Fit Score: {fit_score}/100 ({fit_label})

Key matched keywords: {matched_keywords}
Key missing keywords: {missing_keywords}
Experience alignment score: {experience_alignment_score}/100
Project relevance score: {project_relevance_score}/100
Number of red flags: {red_flag_count} ({high_severity_count} high severity)

Return ONLY a plain text paragraph (no JSON, no markdown).
```

## 10. Scoring Formula (Reference)

```
Total Score (0–100) =
    keyword_component (max 40 pts)
  + experience_component (max 30 pts)
  + project_component (max 20 pts)
  + baseline (10 pts for attempting the role)
  - red_flag_penalty (max -20 pts)

Where:
  keyword_component = (
      must_have_coverage_pct * 0.70 +
      overall_keyword_coverage_pct * 0.30
  ) * 0.40 * 100 / 100   → yields 0–40

  experience_component = experience_alignment_score * 0.30
  project_component    = project_relevance_score * 0.20

  red_flag_penalty = min(
      sum(high×5 + medium×3 + low×1 per red flag),
      20
  )

Labels:
  score >= 70  → "Strong Fit"
  score >= 45  → "Moderate Fit"
  score <  45  → "Weak Fit"
```

## 12. Demo Strategy for Hackathon

### Pre-Demo Preparation
1. **Have 3 pre-computed scenarios ready:** strong fit (student with good match), moderate fit, weak fit. Know which resumes produce which results.
2. **Build a "demo mode" button** that loads a pre-cached result instantly. This way if the API is slow, you can still show the full UI.
3. **Use a real internship posting** (Google, Stripe, or a well-known company). Judges recognize these and it makes the demo feel real.
4. **Have a real student resume** (anonymized or your own). Generic lorem ipsum kills credibility.

### Demo Script (2 minutes)
1. **The hook (15s):** "Students apply to dozens of roles and get no feedback. They don't know if they're a fit, what keywords matter, or how to improve. This tool acts like a career advisor in 30 seconds."
2. **Upload + paste (30s):** Show drag-and-drop resume upload and paste a real JD. Click Analyze.
3. **Show the score (20s):** Show the fit score gauge, the label, and the breakdown. Emphasize "this is not just an LLM guessing — it's a hybrid scoring engine."
4. **Walk through keyword panel (20s):** Show matched (green) vs. missing (red) keywords. Pick one interesting missing keyword.
5. **Show a tailored bullet rewrite (20s):** Show one real example of the original bullet vs. the improved version. This is the most impressive moment.
6. **Red flags + ATS (15s):** Flash these quickly. Emphasize "these are guardrails — we never invent experience."
7. **Close (15s):** "This helps students apply smarter, not harder. We built this in 48 hours."

### Talking Points to Emphasize to Judges
- **Hybrid scoring** (not pure LLM) — shows technical depth
- **Truthfulness guardrail** — shows product thinking, ethics
- **ATS awareness** — shows domain knowledge most students don't have
- **Real-world problem, validated need** — every CS student has felt this pain
- **Modular pipeline** — shows software engineering craft

### Features That Impress Non-Technical Judges
- Animated score gauge / dial
- Color-coded keyword chips (green matched, red missing)
- Before/after bullet rewrite panel
- "Download Report" button (even if it just prints the page)

### Things to Avoid During Demo
- Don't demo on wifi you haven't tested
- Don't show raw JSON anywhere on screen
- Don't demo with a loading spinner longer than 5 seconds (cache a result)
- Don't claim features you haven't built
